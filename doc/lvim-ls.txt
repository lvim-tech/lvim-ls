1. INTRODUCTION                                              *lvim-ls-intro*

lvim-ls is the ENGINE of the LVIM language-server stack: all LSP lifecycle,
project resolution, feature/dependency data and progress tracking — with NO
UI. The visual layer (panels, forms, highlights, notifications) lives in
lvim-lsp, which drives this engine.

The split keeps the heavy, stateful logic isolated and composable: lvim-ls
answers "what servers/tools does this filetype need, what is running, what is
missing", while lvim-lsp decides how to show it and lvim-pkg / lvim-installer
decide how to install it.

No global namespace is touched: every module reads and writes through
|lvim-ls.state|, so the engine stays side-effect free and testable.

==============================================================================
2. ARCHITECTURE                                       *lvim-ls-architecture*

  state.lua          shared runtime state — replaces all _G.* globals
  config/            the live config: lsp / features / progress / message defaults
  data.lua           missing-tool reporting for the unified installer prompt
  health.lua         :checkhealth lvim-ls
  core/
    bootstrap.lua    one-time engine setup (autocmds + initial sweep)
    manager.lua      LSP client lifecycle (start / attach / detach / enable)
    project.lua      project-root resolution and per-project overrides
    schema.lua       per-server settings-form schema resolver (get/set/resolve/apply)
    features.lua     per-buffer LSP feature setup (diagnostics / codelens / on_attach)
    progress.lua     $/progress aggregation (data only)
    dap.lua          registers DAP adapters + configurations from server configs
    globals.lua      JSON persistence of interactively-toggled feature flags
  utils/             notify / debug / levels helpers

==============================================================================
3. INSTALLATION                                       *lvim-ls-installation*

lvim-ls is the engine behind lvim-lsp and is normally installed as its
dependency — you do not configure it directly. Requires Neovim >= 0.10.

lvim-installer (recommended) — install / update / pin it from the Plugins tab of
the LVIM package manager, which installs through Neovim's built-in vim.pack: >vim
  :LvimInstaller plugins
<
Standalone (lazy.nvim): >lua
  {
      "lvim-tech/lvim-ls",
      dependencies = { "lvim-tech/lvim-pkg" },
      -- no config(): a library, configured through lvim-lsp
  }
<
==============================================================================
4. USAGE                                                     *lvim-ls-usage*

lvim-ls is a LIBRARY, not a standalone plugin — there is no `setup()` here. It
is required and configured through lvim-lsp, whose `setup(opts)` merges the
engine (data) options into |lvim-ls.state|.config while the UI options stay in
lvim-lsp. Consumers require the engine modules directly: >lua
  local manager = require("lvim-ls.core.manager")
  local servers = manager.get_compatible_lsp_for_ft("go")
  local missing = manager.missing_tools_for_server("gopls")
<
==============================================================================
5. MANAGER API                                             *lvim-ls-manager*

  local manager = require("lvim-ls.core.manager")

Compatibility / state queries: >lua
  manager.is_lsp_compatible_with_ft(server, ft)   -- boolean
  manager.get_compatible_lsp_for_ft(ft)           -- string[] (incl. "efm")
  manager.is_server_disabled_globally(server)     -- boolean
  manager.is_server_disabled_for_buffer(server, bufnr)
  manager.missing_tools_for_server(server)        -- string[] Mason names
<
Lifecycle: >lua
  manager.ensure_lsp_for_buffer(server, bufnr)    -- attach/start, client id|nil
  manager.lsp_enable(server)                       -- ensure for current buffer
  manager.start_language_server(server, force)     -- start via a compatible buf
  manager.restart_server(server)                   -- stop all, re-attach served bufs
  manager.safe_detach_client(bufnr, client_id)
  manager.stop_servers_for_old_project()           -- DirChanged cleanup
<
A server's binary is resolved at start time as PATH → lvim-pkg's managed bin dir →
the legacy Mason bin dir, so tools installed by lvim-pkg are found without relying
on PATH.
Enable / disable: >lua
  manager.disable_lsp_server_globally(server)
  manager.disable_lsp_server_for_buffer(server, bufnr)
  manager.enable_lsp_server_globally(server)
  manager.enable_lsp_server_for_buffer(server, bufnr)
<
EFM + installation: >lua
  manager.setup_efm(filetypes, tools_config)       -- merge EFM tools, restart
  manager.set_installation_status(boolean)         -- pause/resume auto-start
<
EFM (errorformat) configs are built INTERNALLY from each server module's `efm`
table and aggregated into a single `efm-langserver` client — there is no
external efm.lua. A tool the server needs but Mason has not installed suppresses
the server start; lvim-installer offers the missing tools via the unified prompt.

Per-project overrides live under `.lvim/ls/` at the project root and are managed
through `lvim-ls.core.project`: >lua
  local project = require("lvim-ls.core.project")
  project.list_overrides(root)          -- { servers, filetypes, efm_tools, has_config }
  project.clear_config(root)            -- drop the project feature overrides
  project.clear_server(root, "gopls")   -- drop a per-server settings override
  project.clear_ft(root, "go")          -- drop a per-filetype override
  project.clear_efm_tool(root, "prettier")
<
==============================================================================
6. INSTALLER INTEGRATION                                      *lvim-ls-data*

  local data = require("lvim-ls.data")

lvim-ls.data exposes the missing-tool surface consumed by lvim-pkg's provider
registry, so lvim-installer can offer the missing Mason tools for a filetype
alongside treesitter parsers — without lvim-ls knowing anything about
installation or UI: >lua
  data.missing_for_ft("go")  -- server_name -> { missing dependency names }
  data.items_for_ft("go")    -- flat LvimPkgItem list for the installer prompt
<
==============================================================================
7. BOOTSTRAP                                             *lvim-ls-bootstrap*

  require("lvim-ls.core.bootstrap").init()

`init()` registers the engine autocmds (FileType / BufEnter / BufReadPost
attach, DirChanged cleanup), performs an initial sweep of already-open buffers,
registers the "lsp" requirement provider with lvim-pkg (soft dependency) and
subscribes to its installer events. Activation is deferred by
`config.startup_delay_ms`.
